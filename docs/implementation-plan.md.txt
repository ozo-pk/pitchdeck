PitchDeck — Hackathon Evaluation System
Full Build Prompt (Node.js + Express)

Project Philosophy
Build a database-first web application. Node.js + Express is only a thin bridge between the browser and MySQL. All business logic, validation, scoring, and access control must live inside MySQL using Stored Procedures, Triggers, Views, Functions, and Transactions. No raw SQL in Express routes — every write operation calls a Stored Procedure via mysql2.

Tech Stack

Database: MySQL 8.x (InnoDB)
Backend: Node.js + Express.js
MySQL Driver: mysql2 (promise-based, supports stored procedures and OUT parameters)
Auth: express-session + bcryptjs for password hashing
Frontend: HTML5, CSS3 (Tailwind CSS), Vanilla JavaScript (fetches from Express API)
Environment: .env via dotenv for DB credentials
Dev Server: nodemon for development

db/connection.js — Role-Based Connection Pools
javascriptconst mysql = require('mysql2/promise');
require('dotenv').config();

const pools = {
  admin: mysql.createPool({
    host: process.env.DB_HOST, database: process.env.DB_NAME,
    user: process.env.DB_ADMIN_USER, password: process.env.DB_ADMIN_PASS,
    waitForConnections: true, connectionLimit: 10
  }),
  judge: mysql.createPool({
    host: process.env.DB_HOST, database: process.env.DB_NAME,
    user: process.env.DB_JUDGE_USER, password: process.env.DB_JUDGE_PASS,
    connectionLimit: 10
  }),
  student: mysql.createPool({
    host: process.env.DB_HOST, database: process.env.DB_NAME,
    user: process.env.DB_STUDENT_USER, password: process.env.DB_STUDENT_PASS,
    connectionLimit: 10
  })
};

// Returns the pool matching the logged-in user's role
function getPool(role) {
  return pools[role] || pools.student;
}

module.exports = { getPool };

How to Call Stored Procedures with OUT Parameters (mysql2)
javascript// mysql2 requires a session variable workaround for OUT parameters
async function callSP(pool, spCall, inParams, outParams) {
  const conn = await pool.getConnection();
  try {
    // Set OUT vars to null first
    await conn.query(`SET ${outParams.map(p => `${p} = NULL`).join(', ')}`);
    await conn.query(`CALL ${spCall}(${inParams.map(() => '?').join(', ')}, ${outParams.join(', ')})`, inParams);
    const [rows] = await conn.query(`SELECT ${outParams.join(', ')}`);
    return rows[0];
  } finally {
    conn.release();
  }
}

Express API Routes
POST /api/auth/login

Query SELECT user_id, full_name, role, password_hash FROM users WHERE email = ?
Compare password with bcryptjs.compare()
On success: set req.session.user = { user_id, full_name, role }
Return { role } so frontend can redirect to correct dashboard

POST /api/auth/logout

req.session.destroy() → return 200

POST /api/teams/register (student role)

Call sp_RegisterTeam(hackathon_id, team_name, leader_id, @team_id, @status)
Return { team_id, status }

POST /api/teams/:teamId/members (student role)

Direct INSERT into team_members — this is the only allowed direct write (no SP needed for simple member add)

POST /api/submissions/submit (student role)

Call sp_SubmitProject(team_id, hackathon_id, title, description, github_url, demo_url, @sub_id, @status)
Return { sub_id, status }

POST /api/scores/submit (judge role)

Call sp_SubmitScore(assignment_id, sub_id, criterion_id, score, comments, @status)
Return { status }

POST /api/admin/close-hackathon (admin role)

Call sp_CloseHackathon(hackathon_id, admin_id, @status)
Return { status }

GET /api/leaderboard (public — no auth)

SELECT * FROM vw_Leaderboard WHERE hackathon_id = ?
Return JSON array

GET /api/judge/progress (admin role)

SELECT * FROM vw_JudgeProgress

GET /api/admin/audit (admin role)

SELECT * FROM vw_AuditReport LIMIT 200

GET /api/criteria/:hackathonId (judge + student)

SELECT * FROM criteria WHERE hackathon_id = ?
Used by frontend to dynamically build the scoring form

GET /api/judge/assignments (judge role)

SELECT ja.*, h.title FROM judge_assignments ja JOIN hackathons h ON ja.hackathon_id = h.hackathon_id WHERE ja.judge_id = ?


Middleware — middleware/auth.js
javascriptfunction requireLogin(req, res, next) {
  if (!req.session.user) return res.status(401).json({ error: 'Not authenticated' });
  next();
}

function requireRole(...roles) {
  return (req, res, next) => {
    if (!req.session.user || !roles.includes(req.session.user.role))
      return res.status(403).json({ error: 'Forbidden' });
    next();
  };
}

module.exports = { requireLogin, requireRole };

Triggers (4 Required)
Trigger 1 — trg_AfterEvalInsert: AFTER INSERT on evaluations. Recalculates weighted_total, total_raw, and judge_count in score_aggregates for the affected sub_id using SUM((score/max_score)*weight). Uses ON DUPLICATE KEY UPDATE.
Trigger 2 — trg_AfterEvalUpdate: AFTER UPDATE on evaluations. Same recalculation logic as Trigger 1, fires when admin unlocks a re-score.
Trigger 3 — trg_AuditUserUpdate: AFTER UPDATE on users. Inserts a row into audit_log capturing OLD and NEW values as JSON_OBJECT('email', OLD.email, 'role', OLD.role, 'is_active', OLD.is_active).
Trigger 4 — trg_PreventOpenHackathonDelete: BEFORE DELETE on hackathons. If OLD.status = 'open', raise SIGNAL SQLSTATE '45000' with message 'Cannot delete an open hackathon. Close it first.'

Stored Procedures (4 Required)
SP 1 — sp_RegisterTeam(p_hackathon_id, p_team_name, p_leader_id, OUT p_team_id, OUT p_status)

Check hackathon status = 'open', else return error in p_status.
START TRANSACTION → INSERT into teams → INSERT leader into team_members → COMMIT.
EXIT HANDLER FOR SQLEXCEPTION → ROLLBACK.

SP 2 — sp_SubmitProject(p_team_id, p_hackathon_id, p_title, p_description, p_github_url, p_demo_url, OUT p_sub_id, OUT p_status)

Compare NOW() against hackathons.submission_ddl, set is_late flag.
START TRANSACTION → INSERT into submissions → INSERT placeholder into score_aggregates → COMMIT.
EXIT HANDLER FOR SQLEXCEPTION → ROLLBACK.

SP 3 — sp_SubmitScore(p_assignment_id, p_sub_id, p_criterion_id, p_score, p_comments, OUT p_status)

Validate score between 0 and criteria.max_score.
Validate no conflict_of_interest entry for this judge + team.
Validate hackathon status = 'judging'.
START TRANSACTION → INSERT into evaluations (Trigger 1 auto-fires) → COMMIT.
EXIT HANDLER FOR SQLEXCEPTION → ROLLBACK.

SP 4 — sp_CloseHackathon(p_hackathon_id, p_admin_id, OUT p_status)

START TRANSACTION → UPDATE hackathon status to 'closed'.
CURSOR over SELECT sub_id FROM submissions WHERE hackathon_id = p_hackathon_id.
For each sub_id → INSERT finalization snapshot into audit_log using JSON_OBJECT from score_aggregates.
COMMIT. EXIT HANDLER FOR SQLEXCEPTION → ROLLBACK.


User-Defined Functions (2 Required)
fn_GradeLabel(p_score DECIMAL, p_max DECIMAL) RETURNS VARCHAR(2) DETERMINISTIC
Returns: A+ (≥90%), A (≥80%), B+ (≥70%), B (≥60%), C (≥50%), F (<50%).
fn_IsOnTime(p_hackathon_id INT, p_submitted_at DATETIME) RETURNS TINYINT(1) READS SQL DATA
Queries hackathons.submission_ddl, returns 1 if on time, 0 if late.

Views (4 Required)
vw_Leaderboard — Joins score_aggregates → submissions → teams → hackathons. Uses RANK() OVER (PARTITION BY hackathon_id ORDER BY weighted_total DESC). Calls fn_GradeLabel() per row.
vw_JudgeProgress — Joins judge_assignments → users → hackathons → submissions (LEFT JOIN evaluations). Shows scored count, pending count, completion percentage per judge per hackathon.
vw_TeamScoreDetail — Joins evaluations → criteria → submissions → teams → hackathons. Groups by sub_id, criterion_id. Shows avg score, max score, weight, percentage, grade per criterion per team.
vw_AuditReport — Joins audit_log LEFT JOIN users. Shows COALESCE(full_name, 'SYSTEM'), action, table_name, record_id, old_value, new_value, performed_at DESC.
